# Playground Plan — `try.onlinejourno.com`

Drafted Thu Jun 4, 2026. Targets Wk 12–16 launch (around mid-Aug to mid-Sept).

## Purpose

A read-only community-facing surface where developers, journalists, and observers can see the OnlineJourno product without an account, without live API calls, and without exposing real editorial data.

Three things at once:

1. **Marketing surface** — concrete demonstration of the product, more compelling than a screenshot, less risky than a live tenant.
2. **Code-exploration aid** — readers of the public repo can match the code to the UI it produces.
3. **Brief-output documentation** — shows the brief format, the reasoning-trace format, and the source-attribution model in working state.

## What it is not

- Not a live product. No accounts. No real editorial DNA. No real customer signals.
- Not a sandbox. Users cannot upload content, cannot trigger API calls, cannot interact with agents.
- Not a substitute for the design-partner pilot. Editorial-fit signal comes from the editor, not the playground.
- Not infrastructure that scales. If traffic spikes, content fails gracefully to a static fallback; no auto-scaling, no on-call response.

## What is exposed

- Browseable shortlist of curated sample signals (~50 items, weekly refresh).
- Sample composed brief in the OJ visual identity, in English, dated to last refresh.
- Reasoning trace for each shortlisted item: which source it came from, what the agent said about it, why it was scored as relevant.
- Source registry browser: list of the 25 sources the MVP monitors, with last-seen-at timestamps from real ingestion (the only live data point).
- Read-only views of the journalist UX, editor dashboard, and admin source-config screen. All populated with sample tenant data.
- Architecture diagram + ADR index, linking back to the public repo.

## What is not exposed

- Live Anthropic API calls. The brief and shortlist content are pre-rendered.
- Real editorial DNA from the design partner. The sample tenant uses a fictional newsroom profile.
- The design partner's actual signals, shortlist decisions, or rejected items. None of those data points are visible.
- Account creation, sign-in, payment, billing.
- The admin module-toggle screen for any module the design partner has not yet enabled — modules disabled at MVP stay invisible.

## Content cadence

- **Weekly refresh**, Monday morning IST, founder-driven.
- Curated 50-signal sample drawn from the past week's public regulator releases (RBI, SEBI, NSE, BSE, MCA, etc.) — these are public anyway.
- Sample brief generated by running the production agents against the sample signals in a private dev environment, then exported to static JSON and served from the playground.
- No real-time updates. Last-refresh timestamp always visible in the footer.
- Founder budget for refresh: ≤30 minutes per Monday.

## Sample tenant

The playground uses a fictional tenant called **Dakshin Desk** — a markets / regulatory desk for an imaginary Bangalore-based business publication. Six fictional journalists, English-language brief, default editorial DNA from the standard template.

Why fictional: avoids any implication that a real customer (including the design partner) is endorsing or appearing on the playground without consent.

## Hosting

- **Cloudflare Pages** — static export from Next.js (`next build` then `next export` or static-route equivalent). Free tier covers expected traffic by an order of magnitude.
- Alternative: Vercel free tier (Hobby plan — note licence terms prohibit commercial use, which the playground is not).
- Fallback: tiny Fly.io machine if static export proves impossible. Budget ≤₹300/mo for that path.

DNS: `try.onlinejourno.com` CNAME → Cloudflare Pages (or chosen host).

## Cost

- Static hosting: ₹0/mo expected (Cloudflare Pages free tier).
- Manual refresh API cost: ≤₹100/month (one sample brief generation per week using design-partner-tier model routing).
- Total: ≤₹150/mo. Well inside the ADR 0026 / BRAND-DECISION cap.

## Timeline

- **Wk 12** — design partner has used the brief for 4 weeks; sufficient stability to draft sample content. Begin playground build.
- **Wk 13** — first static export deploys to `try.onlinejourno.com`.
- **Wk 14** — soft-share with three trusted journalists for feedback. Iterate copy and visual polish.

## Refresh procedure (weekly)

1. Mon 09:00 IST — founder collects 50 sample signals from past-week public sources.
2. Run `ingest-score` and `brief-compose` against sample signals in dev environment.
3. Export shortlist + brief JSON.
4. Update sample tenant data in playground static export.
5. `next build && deploy` (CI handles this on commit).
6. Verify by visiting `try.onlinejourno.com`. Check that last-refresh timestamp updated.
7. Log the refresh in a private `playground-refresh-log.md` for trail.

Budget: ≤30 minutes. If a week's refresh would exceed budget, skip the refresh — staleness is acceptable.

## Risks and mitigations

| Risk | Mitigation |
|------|------------|
| Public expects feature parity with described product → disappointment | Footer states explicitly "Read-only demo. Account-backed product runs at app.onlinejourno.com from Wk 8." |
| Sample content is misleading or factually wrong about regulators | Founder reviews refresh before publish. Source links go to original regulator pages so readers can cross-check. |
| Demo perceived as the actual product → false negative reviews | Stage-banner across the top: "Demo with sample data — refreshed weekly". |
| Refresh skipped for several weeks → looks abandoned | Footer shows last refresh; if >14 days, automatic banner: "Founder away — see GitHub for updates." |
| Hosting service outage during launch | Static export means no dependency on origin server. Cloudflare Pages SLA acceptable for a free demo. |
| Cost spike from unexpected viral traffic | Cloudflare free tier handles ~100K req/day. Beyond that, traffic rate-limit at CDN; no auto-scaling, no surprise bill. |
| Demo content becomes accidentally entered into search engines as if real | `robots.txt` allows indexing but every page carries a "Sample data — demonstration only" meta description and visible label. |

## Decommission plan

The playground may be paused or shut down with 30 days notice on the homepage. Sample data is not personal data, so no data deletion process beyond ceasing the static export is needed.

If shut down: redirect `try.onlinejourno.com` to a `marketing/demo-paused` page on the main marketing site.
